# 学生情绪评估与预警体系（开发文档）

本文档描述 **student-emotion-web 管理端** 与同源库 **EmoDetect-DigitalMan 学生端** 中，与「情绪采集 → 标准化 → 可视化 → 预警 → 处置」相关的**权威定义、数据口径与实现位置**。实现以代码为准；若产品与代码不一致，以本文档驱动迭代时须同步改代码与 UI 文案。

---

## 1. 术语与分层模型

| 概念 | 定义 | 典型来源 |
|------|------|----------|
| **原始采集** | 单次识别或交互产生的一条记录 | `emotion_record`（人脸）、历史 `sem_emotion_point`、数字人侧信号等 |
| **情绪标签（细粒度）** | 模型或规则输出的类别字符串，如 `sadness`、`悲伤` | `emotion_record.emotion_type` |
| **强度 intensity** | 多为模型置信度或 0–1 强度，**不等价于心境三分类** | `emotion_record.intensity` |
| **标准化分数 score** | 归一化到 **0–100** 的整数，用于趋势、等级区间、部分统计 | 由 `_score_from_intensity` 计算 |
| **心境 mood（三分类）** | **积极 / 中性 / 消极**，用于分布饼图、预警条数统计 | 由 `_mood_from_emotion_label` 从细粒度标签映射 |
| **危险等级 level** | **低 / 中 / 高 / 危**，用于预警卡片展示 | 由 `_risk_level_from_score` 对照管理员配置的分数区间 |
| **预警 alert** | 对辅导员可见的一条待处置事项，含状态流转 | `sem_alert` 表 + 运行时合成记录 |
| **集团可视化** | 对「当前登录人可见的全部学生」做聚合，**不是**单生日志回放 | `compute_visualization` |

**关键原则（必须写进产品说明）：**

- **score 与 mood 独立推导**：同一条记录可以「标签映射为消极」，同时「强度换算分数仍较高」（例如高置信悲伤 → 消极心境 + 高分）。折线图看 **score**；饼图「情绪分布」看 **每人最新一条的 mood**；预警里的「近 7 天消极次数」看 **mood**。
- **可视化里的「今日平均」**：当前实现为「可见学生每人取时间线**最后一条**的 score，无数据记 50，再算术平均」——对无数据学生会拉高中位数，属已知口径，见 §5.2。

## 2. 原始数据与时间线

### 2.1 时间线 `get_emotion_timeline(student_no)`

实现：`student-emotion-web/backend/database.py` → `get_emotion_timeline`

**读取顺序（管理端口径）：**

1. 若存在表 `emotion_record`，且该学号在 `emotion_record` 中有行：  
   - 仅使用 `emotion_record`（按 `timestamp` 升序）。  
   - 来源字段展示为「人脸识别」。
2. 若存在表 `emotion_record` 且存在核心表 `student`，且该学号在 `student` 中存在、但 `emotion_record` **为空**：  
   - 返回 **空列表**（**不**回退到 `sem_emotion_point`，避免已删学生孤儿数据污染）。
3. 否则回退：`sem_emotion_point`（兼容旧数据）。

每条时间线点结构（逻辑上）：

- `ts`：毫秒时间戳  
- `score`：`int`，0–100  
- `mood`：`积极` | `中性` | `消极`  
- `source`：如 `人脸识别`

## 3. 标准化规则（权威算法）

### 3.1 分数 `_score_from_intensity(intensity)`

位置：`database.py`

| 条件 | score |
|------|--------|
| 无法解析为浮点 | `50` |
| `intensity ≤ 1` | `round(clamp(intensity × 100, 0, 100))`（常见：人脸置信度 0–1 → 映射为 0–100） |
| `intensity > 1` | `round(clamp(intensity, 0, 100))`（兼容已放大的百分制） |

**含义：** score 表示「强度在量表上的位置」，**不**直接表示「积极/消极」语义；语义由 §3.2 决定。

### 3.2 心境三分类 `_mood_from_emotion_label(label)`

位置：`database.py`

- 将 `label` 转小写字符串后与**固定词表**匹配：  
  - 命中 **积极词表** → `积极`  
  - 命中 **消极词表** → `消极`  
  - 否则 → `中性`  
- 词表包含常见英文（如 `sadness`, `anger`）与中文（如 `悲伤`, `焦虑`）。**未在词表中的标签一律中性**（扩展时需改代码并回归测试）。

### 3.3 危险等级 `_risk_level_from_score(score, levelRules)`

位置：`database.py`

- `levelRules` 来自表 `sem_threshold` 的 `level_rules_json`，由管理员在「预警阈值设置」页维护。  
- 每条规则包含 `minScore`、`maxScore`、`level`（`低|中|高|危`）。  
- 按规则表顺序匹配第一个满足 `minScore ≤ score ≤ maxScore` 的区间；若无命中则默认 **`中`**。

### 3.4 敏感度 `sensitivity`

- 存储于 `sem_threshold.sensitivity`（0–100），**当前版本未参与**时间线、可视化或 runtime 预警的数值计算（预留与产品扩展）。  
- 开发文档需标注：**未接线字段**，避免与「算法已生效」混淆。

## 4. 预警体系

### 4.1 两类记录

| 类型 | 说明 |
|------|------|
| **持久化预警** | 表 `sem_alert`，含指派辅导员工号、状态、备注等 |
| **运行时预警（runtime，合并）** | 每人 **一条** 虚拟/落库记录，**固定 `id = runtime-{学号}`**。近 7 天内 **`人脸/时间线消极条数 + 聊天情绪异常条数（emotion_anomalies，from_monitoring=0）` 合计 ≥ 3** 时出现；**当前档 `k = 合计 // 3`**。列表拉取时若该 `id` 已在库中且状态为 **NEW/FOLLOWED**，会 **UPDATE** 等级、原因、时间戳以反映升档；**不再**分别展示 `runtime-chat-*` 与 `runtime-*-xN`（旧 id 在列表中会被过滤，历史行可仍在库中直至人工处理）。`materialize_runtime_alert` 仅写入/识别 **`runtime-{学号}`**（兼容旧请求 URL 中带 `runtime-chat-…` / `-xN` 时解析学号）。 |

### 4.2 生成规则（runtime，合并口径）

位置：`database.py` → `_unified_signal_counts`、`_unified_runtime_alert_item`、`list_alerts_for_staff`

对辅导员 **可见学生**：

1. **人脸/时间线**：近 7 天、`mood == 消极` 的条数记为 `n_face`。  
2. **聊天**：`emotion_anomalies` 与 `users` 按学号对齐、`from_monitoring=0`、近 7 天的条数记为 `n_chat`。  
3. **`total = n_face + n_chat`**；若 `total < 3`：不生成。  
4. 否则 **`k = total // 3`**，生成/更新一条：`id = runtime-{学号}`，`reason` 中写明 `n_face、n_chat、total` 与当前档 `k`；**等级**与档位一致：`_unified_alert_level_from_gear(k)`（`k≤1` 低、`k=2` 中、`k=3` 高、`k≥4` 危），**不再**用人脸时间线分数套 `levelRules`，避免「仅几条聊天、文案写第 1 档却显示高」的口径冲突。  
5. **聊天异常**仍依赖数字人侧大模型判异常并落库；仅口头「不开心」未落库则不计入 `n_chat`。

### 4.3 状态与权限

- 状态：`NEW`（待处理） / `FOLLOWED`（已跟进） / `CLEARED`（已消除）。  
- 更新接口：`POST /api/counselor/alerts/:id`，权限校验见 `app.py`（辅导员仅能处理指派给自己或 `SYSTEM` 且学生在自己范围内的预警等）。

## 5. 集团可视化（辅导员/管理员首页）

实现：`database.py` → `compute_visualization`  
接口：`GET /api/counselor/visualization?range=week|month|term`  
前端：`DashboardPage.tsx` / `VisualizationPage.tsx`

统计总体均为：**当前角色可见学生集合 `visible_students(staff)`**。

### 5.1 情绪分布比例（饼图）

- **单位：学生人数**（不是识别条数）。  
- 每个学生：取时间线 **按时间排序后的最后一条**；若其 `mood ∈ {积极,中性,消极}`，则对应计数 +1；**无时间线**则计为 **中性**。  
- 饼图百分比 = 该类人数 / 三类人数之和（即可见学生中有时间线的人数与「无数据算中性」的人数之和）。

**与预警的关系：** 某生可因「7 天内 ≥3 条消极」出预警，但饼图只反映其**最新一条**心境，故「预警多」不一定「消极占比大」。

### 5.2 今日群体平均 `todayAvg`

- 每个学生：有最后一条则用其 `score`，**否则用 50** 参与平均。  
- **含义：** 近似「当前群体最近一次识别的平均强度位置」，**不是**官方量表临床分。若需「仅对有数据学生平均」，应改代码并同步文档。

### 5.3 趋势折线 `trend`（周 / 月 / 学期）

**分桶时区：** 环境变量 `SEM_DISPLAY_TZ`（默认 **`Asia/Shanghai`**），与学校业务日历一致。

**对每个自然日 D（在展示时区内）：**

1. 计算该日 `[当天 0:00, 次日 0:00)` 的毫秒区间 `[t0, t1)`。  
2. 对每个可见学生，取其时间线中落在该区间的所有点的 **score**，若有则取 **算术平均** 作为该生当日的代表分；该日无任何点则该生**不参与**当日分子。  
3. **群体日均**：对所有「当日有数据」的学生的代表分再取 **算术平均**；若当日无人有数据，则 `avg = 0`。

**折线横坐标：** 每个点的 `ts` 取该自然日起点 `t0`（毫秒），前端格式化为 `MM-DD`。

**与旧版差异说明：** 历史实现曾用「锚点时刻 ± 找整条时间线上时间最近的一点」，易产生与日历直觉不符的波动；当前以 **自然日 + 日内平均** 为规范口径，便于写进验收说明与论文式方法描述。

## 6. 管理端首页其他指标（与 `compute_visualization` 并列）

实现：`DashboardPage.tsx`（前端聚合）

- **今日新增预警数**：`counselorListAlerts` 返回的预警中，`createdAt ≥ 本日 0:00（浏览器本地）` 且 `status === 'NEW'` 的条数。  
  - 注意：runtime 预警的 `createdAt` 取自后端规则（见持久化/materialize 代码），与「插入数据的时刻」不一定相同。  
- **高危学生 Top 3**：对可见学生列表与预警列表做关联：每个学生取**未消除**预警中的**最高**等级，再映射为「高危/中危/低危/正常」标签后排序取前三（具体映射见前端 `mapAlertLevelToRiskTag`）。

## 7. 学生端（EmoDetect-DigitalMan）补充口径（同源库）

以下不在 `student-emotion-web` 内计算，但与同一 `emotion_record` 表相关，开发文档建议**交叉引用**：

| 能力 | 说明 |
|------|------|
| **人脸异常标签集合** | `FACE_ABNORMAL_EMOTIONS`：`anger`, `contempt`, `disgust`, `fear`, `sadness`（小写匹配） |
| **近 7 天人脸异常次数** | `count_recent_face_abnormal_records`：用于主动疏导阈值统计 |
| **主动疏导** | 与聊天侧异常次数合并计数，达阈值写入 `proactive_triggers`；与辅导员端「预警列表」规则**独立** |

学生端与管理端可共用 `emotion_record`，但**业务目的不同**：一端偏「即时干预与对话」，一端偏「管辖范围内群体态势与台账」。

## 8. 数据权限（影响一切聚合）

辅导员可见学生由 `within_scope` 判定：学院、年级、专业、`class_id` 必须与其 `scope_json`（或 `scopes` 多段）之一匹配。  

**结论：** 同一学号上的情绪记录，仅对**管辖范围包含该生**的辅导员出现在可视化与预警中；换账号「看不到」多为**权限非 Bug**。
